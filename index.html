CI in an Agentic World

How AI agents will transform continuous integration

Introduction

The world is changing rapidly nowadays with the introduction of “AI” — or more accurately, the introduction of coding agents and agentic IDEs (Claude Code, Cursor, Codex, etc.).

In this article I’m going to try and ideate about the future of CI, agentic coding, validations and automation of distributed agents.

The AI Shift

Change examples

The world is changing drastically. Anthropic showed recently that they’ve managed to write a C compiler with a team of parallel Claudes:

“16 agents writing a Rust-based C compiler, from scratch, capable of compiling the Linux kernel. Over nearly 2,000 Claude Code sessions and $20,000 in API costs, the agent team produced a 100,000-line compiler that can build Linux 6.9 on x86, ARM, and RISC-V.”

Or businesses run only by distributed agents, such as Paperclip (orchestrator) or spec-driven development (spec-kit) which allows humans to define the project as specs and then later orchestrate AI agents in order to implement it.

Code reviews, GitHub issue reviews, fully distributed agentic flows are becoming a must in repos and become a live engine. For example, Anthropic’s GitHub Actions in their own Claude Code repo.

This shift shows that who really wrote the code is not human — the human is solely the orchestrator, and their main goal is to create a safety net that validates that their intent was executed correctly while keeping the past intent as-is.

The example also shows that with good infrastructure, you can parallelize 16 agents with validations and communication between them, which in turn allows us to write code autonomously with high confidence.

Volume explosion

Agentic coding (Cursor, Claude Code, Copilot Workspace) means 10–100x more PRs/commits per engineer. CI infrastructure that takes 20 min/run breaks economics immediately.

Trust model inversion

Human commits were implicitly trusted for intent. Agent commits need a new trust tier: the CI system becomes a verifier of agent output, not just a build runner.

Tests as the spec

When agents write both code and tests, the spec must live elsewhere. CI needs to validate against intent (requirements, design docs) — not just pass/fail. Evals become a CI primitive.

Flakiness becomes critical

A human can re-run a flaky test and use judgment. An agent in a loop will spin indefinitely or make wrong corrections. Flaky test elimination becomes a hard blocker.

The feedback loop IS the product

For agents, CI feedback quality (not just speed) determines output quality. Vague “build failed” messages cause agents to hallucinate fixes. Structured, semantic error signals are now a feature.

Dynamic test selection

Static “run all tests in affected packages” is too blunt. AI-powered test selection (e.g., Launchable, Currents) predicts which tests to run based on code semantics, not just file proximity.

How CI Architecture Should Look

Continuous testing, not triggered testing

Decouple “something changed” from “run validation.” Tests run against the semantic graph continuously. A change gets a “confidence score” rather than pass/fail.

The build graph as a live service

Not computed per-run but maintained as a living graph (like a database). Incremental updates as files change.

Agent-aware merge queues

Merge queues (like Trunk, Aviator, Mergify) need to understand agent provenance. Agent PRs can be batched differently, tested more aggressively on evals, and auto-retried on infra failures.

Semantic caching

Beyond content-addressed caching; cache at the level of “did the observable behavior of this module change?” Type-level and runtime-behavioral equivalence as cache keys.

CI as an AI feedback oracle

The CI system actively returns structured feedback to the agent: which assertions failed, what the type error means, what similar past failures looked like. Not logs — structured signals.

Sandboxed agent environments

Each agent run gets an ephemeral, reproducible environment. Think: dev containers + hermetic builds + snapshot testing of side effects. The environment is part of the artifact.

Reviewing Code Becomes Harder Yet More Critical

When agents produce 10–100x more code, the human reviewer becomes the primary quality gate. But reviewing agent-generated code is fundamentally different: changes are larger, context is less familiar, and the reviewer can’t rely on “I know what the author was thinking.” Code review needs to evolve from line-by-line diffing to intent verification — and the tooling must keep up.

Semantic code reviewing

Current diffs show every changed line in file order, mixing type signatures, business logic, documentation, and formatting into one undifferentiated stream. Semantic review tools should let reviewers filter by category of change: show only logic changes, only type changes, only comment/doc updates. This turns a 500-line diff into several focused, reviewable slices — each one answerable to a clear question (“did the types stay consistent?” vs. “does the logic match the spec?”).

Stacked diffs

A single large change can be decomposed into a chain of dependent PRs, each small enough to review meaningfully. Tools like Graphite and ghstack keep the entire stack rebased and in sync automatically. This reduces CI cost (each layer runs only its own tests) and makes review tractable — reviewers focus on one semantic step at a time rather than the whole transformation. See stacked diffs vs. trunk-based development for trade-offs.

Purpose-built version control

Git was designed for human workflows: linear history, manual conflict resolution, branch management. Agent workflows need cheap forking, automatic rebasing, and conflict-free concurrent edits. Tools like jj (Jujutsu) offer first-class support for these patterns — anonymous branches, operation log for undo, and a simpler mental model that agents (and humans) can work with more reliably.

Optimized worktree workflows

Agents working in parallel need isolated working copies. Git worktrees provide this, but each worktree traditionally requires its own node_modules install and cold build cache. Using content-addressable package managers like pnpm (which hard-links shared dependencies) and shared build/test caches across worktrees eliminates the per-worktree overhead, making parallel agent execution practical without multiplying resource costs.
